- `SELECT COUNT(*) FROM tool_usage_events` — **0 rows**

Sink was clearly not exploding (no soft-warning logs); rows were not
silently rejected (slug-not-found would have warned); request was healthy
end-to-end (200 OK + counter bump). Smoking gun: writes happened then
disappeared. Pointed straight at session-commit lifecycle.

### Root cause

The `/recommend` request path writes `ToolUsageEvent` rows then never
commits the SQLAlchemy session; rows are rolled back when `get_db` closes.

Trace:

1. `RecommendationService.recommend` calls `self.emit_usage(...)` at
   `core/recommend/service.py:229` for each in-catalog rec — *firing*
2. `emit_usage` is `make_db_sink(db)` wired at
   `core/api/recommend.py:115` — *correctly wired*
3. `emit_usage_event` at `core/telemetry.py:147-148` does
   `session.add(evt)` + `session.flush()` and **does not commit**.
   Docstring at lines 116-126 explicitly states the contract: *"The
   caller owns the transaction: this helper adds + flushes but does
   not commit. Commit boundaries stay with the service / endpoint
   layer that owns the Session."*
4. Endpoint at `core/api/recommend.py:122-128` returns the response
   without ever calling `db.commit()` — *the bug*
5. `get_db` at `core/db/session.py:30-35` yields the session and only
   `session.close()` in `finally` — no implicit commit
6. Session factory `autocommit=False, autoflush=True` (line 27) —
   closing without commit rolls back the implicit transaction; the
   flushed `ToolUsageEvent` row is discarded

### Why install works but recommend doesn't

The install path emits at `core/lifecycle_store/service.py:538` then
calls `self.update_status(...)` at line 556 → which commits at line
394. **Same session.** The flushed telemetry row rides along on the
lifecycle commit. The sink contract was designed assuming the caller
commits later — true for install (lifecycle commits downstream), false
for recommend (nothing does).

### Why existing tests passed

`tests/test_recommend_service.py::TestUsageTelemetryWiring` (lines
484-588) uses `_CapturingSink` — an in-memory list. Validates *the
service calls the sink with the right args*. Never exercises
`make_db_sink` against a live session; never asserts a committed row
exists. The bug lives one layer above the service, in the FastAPI
endpoint's session-lifecycle wiring, which has no integration test
that reads from the DB after the request. Classic wiring-test gap:
each piece is unit-tested correctly; the full request → committed-row
chain is unverified.

### Three options considered

- **Option A (chosen)** — Fix in endpoint, ~3 lines: add
  `db: Session = Depends(get_db)` to the recommend endpoint, call
  `db.commit()` after `service.recommend(req)` succeeds. Keeps the
  "caller owns the transaction" contract intact; matches the install
  pattern (downstream code commits); explicit at the read-the-code level.
- **Option B (rejected)** — Commit inside `make_db_sink`'s closure,
  ~2 lines. Rejected: makes telemetry committable in isolation from
  the triggering operation. If the recommend handler later fails after
  the emit (e.g. fixture-drift validator gains a hard-raise variant in
  a future change), a phantom-event row would be left behind. Telemetry
  should ride with the operation that produced it, not commit
  independently.
- **Option C (rejected)** — Commit-on-success in `get_db`'s finally,
  ~3 lines. Rejected: silently changes transaction semantics across
  every endpoint in the codebase. Any future endpoint that thought it
  was rolling back via "just don't commit" would suddenly commit
  partial writes. The kind of change that causes mystery data
  corruption months later when no one remembers it landed.

### Severity

Real bug, **non-blocking for Fix Day 4 ack-as-complete**. Reasoning:
the user-facing pattern (narration) validated end-to-end. The
data-substrate pattern (telemetry rows persisted across requests) is
broken on the recommend path but works on the install path. The 34
demotion-candidate baseline is a real signal source already; missing
recommend-event signal degrades scanner accuracy for tools that get
recommended-but-never-installed-or-loaded — recoverable once Task 0
of Day 5 lands.

Bug doesn't worsen overnight; no data corruption (rolled-back rows
just don't exist); no user-visible failure mode in the live shakedown
beyond the empty-table observation.
