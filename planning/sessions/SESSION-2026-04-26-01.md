## Appendix E — Live shakedown discipline (codification proposal)

Surfaced as a meta-lesson this session. Documented here pending Lewie's call on Open Question 2 (codify in `docs/concierge-operations-protocol.md` vs feedback memory vs both).

**Rule:** Live-shakedown verification of a fix MUST run from a fresh Claude Code session that has not been exposed to the build conversation that produced the fix. Build sessions encode the bug + fix narrative in their context window; their narration of a successful tool call is contaminated evidence — it could equally explain "the fix landed" or "the agent inferred the expected output from build context." Fresh sessions strip this confound: the agent reasons solely from the user's prompt and the production-side state (catalog, memory, MCP resources).

**Why it matters:** The technical signal (DB rows landing, counters incrementing, scanner counts shifting) is unaffected by which session triggered the call. But the user-experience claim ("the agent invoked Concierge correctly and narrated the consultation") cannot be validated against a session whose context already names the bug, the fix, and the expected behavior. Fresh-session is the only honest test.

**This session's mistake:** Lewie initially typed the apples-to-apples shakedown prompt into THIS build session by mistake. The recommend call DID fire (`recommend.requests` went 0→1, rows landed in DB), but the validation argument was contaminated. We restarted with a fresh Claude Code session, baselined the DB to 0 rows, and ran the prompt cleanly. Technical signal: 0→4 rows, identical to the contaminated run modulo timestamps. Validation argument: now clean.

**Discipline statement (proposed for ops-protocol):** "Before triggering a live-verify shakedown, confirm the target Claude Code session is fresh. If you must reuse a session, document explicitly that the verify is technical-signal-only and the user-experience claim requires a separate fresh-session pass."

### Task 1 — Streaming wire-format test resurrection + bug surfaced + bug fixed

**Commit:** `6f5239f` `fix(events): JSON-serialize SSE data field at handler boundary`. Single coherent commit per the same discipline as Task 0 (`8ee0af3`). Sequencing: the original "test resurrection" deliverable is exactly what surfaced the bug it was designed to catch on the first run; commit ships test + fix together rather than splitting.

### Bug summary

The SSE handler at `GET /ui/events` emitted `data:` lines that weren't JSON. The broker held raw Python dicts; sse-starlette's default `str()`-on-non-string serialization produced Python repr (single quotes, `None`/`True`/`False`):

```
event: new_request
data: {'filename': '...', 'category': None, 'is_discovered': False}
```

`JSON.parse(event.data)` in HTMX (and any standard SSE consumer) would reject this with SyntaxError. Day 4's "manual curl validated" check confirmed bytes flowed but never asserted JSON-ness.

### What surfaced it

The streaming wire-format test resurrection itself. The first end-to-end run produced the chunk above (Python repr) — exactly the format that fails `JSON.parse`. The test's regex extraction worked; my added explicit `json.loads(raw_data)` step (per Lewie's discipline-strengthening point) is what made the failure unambiguous.

Test discipline carry-forward: a regex-only test that asserts field names without `json.loads` would have *passed* on Python repr (since the field names happen to appear in `{'filename': '...'}` too). The explicit JSON parse + re-pin of `data["category"] is None` / `data["is_discovered"] is False` are the load-bearing assertions — they fail on Python repr because `json.loads` rejects single-quoted dicts and capitalized booleans.

### Audit finding

Layer-1 grep for `EventBroker.publish` call sites: **one site today** (`core/lifecycle_store/service.py:258`, `new_request` event with dict data — the bug). No other event types fan out through the broker. Handler-layer fix (4 lines in `core/api/events.py`'s `_stream` async generator with `isinstance(data, (dict, list))` guard → `json.dumps`) protects all future event types by construction; no per-call-site changes needed.

Reasoning for the location choice: the broker is a transport-agnostic fanout designed to feed future non-SSE consumers (WebSocket bridge, log tee, in-process tap) raw Python objects so they don't have to parse SSE bytes. Pre-serializing at the publish site would couple the broker to the SSE wire format and break that design. The handler is the right boundary — it knows about SSE, the broker doesn't.

### Test additions

- **`TestStreamingWireFormat::test_new_request_event_appears_on_wire_with_correct_format`** (new). Runs a real uvicorn server in a background thread on a free localhost port. Env-overridden DB + lifecycle paths so no production state is touched. The in-process `httpx.AsyncClient + ASGITransport` path explored on Day 4 and re-explored Day 5 (with `pytest-timeout` + thread-method timeout in place) always hangs because `ASGITransport.handle_async_request` awaits the app's `__call__` to fully complete before yielding the response — and `EventSourceResponse` is infinite by design. Real HTTP via uvicorn-in-thread is the only reliable transport (the "ASGI-LIFESPAN-aware harness" alternative today.md acknowledged).

- **Strengthened JSON-correctness assertions** per Lewie's point: explicit `json.loads(raw_data)` (not just regex extraction) catches Python-repr-of-dict-with-string-values, which would silently parse-fail in subtle ways under a regex-only test. Re-pin: `data["category"] is None`, `data["confidence"] is None`, `data["is_discovered"] is False` — these specifically catch repr-style `None`/`False` regressions because `json.loads` rejects them.

### Meta-lesson — wiring tests must assert client-observable contracts

Two consecutive days, two bugs of identical shape:

- **Day 5 Task 0 (telemetry-commit):** existing tests asserted "service called the sink with the right args" (in-memory `_CapturingSink`). They did NOT assert "row exists in the DB after the request." End-to-end persistence chain unverified. Bug shipped.
- **Day 5 Task 1 (SSE wire format):** existing tests asserted "broker received the event dict with the right fields" (broker-direct + service-publish coverage). Day 4's manual curl asserted "bytes flowed on the wire." Neither asserted "the bytes parse as JSON the way an HTMX client would parse them." Bug shipped.

The pattern: layered unit tests pass; the integration-level claim a client actually depends on (persisted row count for telemetry; JSON parseability for SSE) is unverified. **Lesson: wiring tests need to assert client-observable contracts, not just whether bytes/calls flowed.** Concretely:

- If a write feeds downstream consumers, the wiring test reads from a fresh session/transport boundary and asserts the consumer-visible state changed
- If a serialized payload feeds a parser, the wiring test deserializes via the same parser the client would use (`json.loads` for HTMX SSE; `urllib3.HTTPResponse` decoding for HTTP body schemas; etc.)
- If a counter increments, the wiring test reads the counter from the same surface the operator would (`/health` endpoint, not the in-memory object)

This generalizes beyond Day 5. Should land as a discipline note in `docs/concierge-operations-protocol.md` (under "Test fixture management" or as a peer "Test discipline" section) — flagged as Open Question for Day 6 / pre-public-push prep window.
