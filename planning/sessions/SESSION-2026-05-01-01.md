### Real defect surfaced during live-verify

**Approve button has no in-flight UX feedback during the ~8 second backend processing window.** Operator clicked Approve, button-press visually acknowledged, then 8 seconds elapsed before the card disappeared. Operator clicked at least twice during the gap to ensure the click registered.

Two facets:

1. **UX gap (essential fix).** No `hx-indicator` or button-disabled-during-request behavior. Standard HTMX pattern shows a spinner + disabled state during in-flight requests. Without it, users in production will double-click on slow actions (which is exactly what happened). Backend correctly handles double-submission via InvalidTransitionError → 409 Conflict, but user has no signal of either the in-flight state or the duplicate-rejection state. **Day 11 polish work**; not blocking Day 10 close.

2. **Performance question (investigation, not fix).** Why does approve take 8 seconds? Possibilities: DB write + file rename (likely fast), downstream side effects (autonomous install attempt? scanner re-scan? memory write?), synchronous external API call somewhere in lifecycle service. Forward-carry; not Day 10 or Day 11 priority unless demo recording surfaces it.

### Counter inconsistency observation (design call, not bug)

**Health/Stats counter showed "Pending requests: 1" while inbox showed "No pending tool requests."** Investigation:

- Health/Stats counter (`core/api/health.py:_requests_counts_by_folder`) counts rows by `folder` only.
- Inbox (`core/lifecycle_store/store.py:list_pending_rows:193`) filters `folder='pending' AND status='pending'`.
- DB state at observation: 2 rows in `folder=pending` with non-pending statuses — id=6 (csvkit, status=failed since 2026-04-23) and id=7 (ocrmypdf, status=installed after operator's approve action). Both stuck in `pending/` folder until cron picks them up.

**Documented behavior** per `list_pending_rows` docstring ("the cron moves files based on status, so any mismatch between folder and status is a signal the reconciliation pass is out of date") — but the user-visible inconsistency between Health bar and inbox is real and confusing. **Day 11 design decision** required: this is a real semantic question (does the bar show pending-only matching the inbox, or all unresolved requests including deferred?), not a bug to quietly patch in close-out. Quietly aligning would foreclose the design conversation; surface in Day 11 alignment for the right semantic.
