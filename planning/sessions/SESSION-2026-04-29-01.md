## Appendix D — Clean-baseline-regression-signal discipline evolution

Day 7 ratified the clean-baseline-regression-signal discipline:

> "The suite is now at clean baseline; any regression from this point is unambiguous signal. Treat new failures with full surface-investigation discipline before assuming flakiness. The 'passes alone, fails under suite' attribution shorthand from Days 5-6 is now a known wrong-cause-attribution shape; new failures get full diagnosis, not categorization."

Day 8 produced an unexpected refinement.

### What happened

Day 7 close-out's measurement was: 783 passed, 0 failed (single full-suite run). The discipline ratified on this measurement said "any regression from this point is unambiguous signal." Day 8's first full-suite run produced 13 failures; second run produced 0 failures with same code state.

If the Day 7 close-out measurement had been N=5 samples, the failure mode would have surfaced as 1/5 = 20% flake rate at Day 7 close, NOT as a clean baseline. The "0 failures" was a true measurement at Day 7 close; it was just a single-sample measurement of a failure mode that's intermittent.

### The refinement

The discipline applies as written — investigate before categorizing as flakiness — but the practical implication is:

**Single-shot suite runs aren't proof of clean baseline if a failure mode is intermittent. Flake-rate characterization (N≥5 samples) is the stronger signal.**

This doesn't invalidate the Day 7 ratification; it adds a measurement-discipline layer underneath. The Day 7 discipline says "investigate, don't categorize"; the Day 8 evolution says "your evidence for 'clean baseline' should be N≥5 samples, not a single run, when the test surface includes integration tests with non-deterministic behavior."

### Application to Day 8

Today's flake-rate investigation (Appendix B) followed the refined practice: single failure observed → falsifying-hypothesis test (Phase 0 lockfile drift) → narrowing test (Phase 1 targeted-files re-run) → quantitative characterization (Phase 1 Step 2 N=5 sampling). Surface findings produced WITHIN time-box, not lazy-categorization OUTSIDE time-box.

### Candidate for ratification

This refinement is **flagged as candidate** for ops-protocol ratification, NOT ratified today. Per the same maturation discipline (await second data point before ratifying), a future occurrence — specifically, another build day where the suite shows zero failures on first run but produces some non-zero rate under N=5 sampling — would be the second data point.

If ratified, the discipline language becomes:

> "Clean baseline is a claim about the test surface's behavior, not a property of a single run. For test surfaces that include non-deterministic-prone tests (integration / network / heavy fixture / real-subprocess), the evidence standard for 'clean baseline' is N≥5 samples produced by the same code state, not a single full-suite pass."

Until ratified, today's refinement lives in this Appendix as the second occurrence of flake-rate-mattering-for-baseline-claims (Day 5/6 was the first).
