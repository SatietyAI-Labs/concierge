### Durable lesson codified

Memory file written: `feedback_public_contact_info_confirmation.md`, indexed in `MEMORY.md`. Rule: never infer personal contact info (email, phone, mailing address, social handle, real-name attribution) from git config / env vars / auto-memory when proposing values for public artifacts. Always ask the operator first.

Threshold for "is this a public-artifact value?": *would this value, once committed to a public repo, be visible to anyone on the internet forever?* If yes, ask.

### Why this matters

Personal/operational addresses leaking into permanent public artifacts is hard to reverse:
- Versioned in git history
- Indexed by package registries (PyPI scrapes pyproject metadata)
- Crawled and cached by aggregators (libraries.io, GitHub search, archive.org)
- Single accidental commit can put the address in long-tail public records permanently

This event prevented that for the Concierge package metadata. The operator's broader framing post-event: *"For future reference: when proposing personal contact info in public artifacts, always confirm with the operator rather than inferring from git/env config. Local config defaults aren't authorization for public exposure."*
