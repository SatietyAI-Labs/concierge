### Manual verification arc — four bugs surfaced and fixed

The `claude mcp add-json concierge '{"command":"/abs/path/scripts/concierge-shim"}'` registration succeeded. Launching a fresh Claude Code session in a new terminal and invoking `concierge_recommend` surfaced bugs iteratively. Each fix landed as its own commit with a regression guard before the verification could advance to the next step.

1. **Pydantic venv path bug** (commit `e8a7e1b`). `scripts/concierge-shim` shebang was `#!/usr/bin/env python3`, which resolved to `/usr/bin/python3` under Claude Code's MCP spawn env (no venv activation). System Python lacks Concierge deps → `ModuleNotFoundError: No module named 'pydantic'`. Fix: absolute-path shebang `#!/home/satiety/.venvs/concierge-hackathon/bin/python3`. Venv's python3 is a symlink to /usr/bin/python3 on this host; the venv site-packages discovery relies on the INVOCATION path (argv[0]), so the absolute path keeps that working. Regression test `TestWrapperScriptInvocation::test_wrapper_spawns_without_venv_activation` spawns the wrapper directly (not via `sys.executable` like every existing e2e test) with scrubbed `PATH=/usr/bin:/bin` — explicit assertions on ModuleNotFoundError absence + `shim.startup` presence. This test is the first to exercise the actual executable-invocation path; all 25 pre-existing e2e tests bypassed the wrapper via `sys.executable`.

2. **ProtocolVersion mismatch bug** (commit `bc88327`). Shim advertised `2025-11-05`, real Claude Code 2.1.117 rejected with `Server's protocol version is not supported` — closing the MCP connection. **Root cause: transcription typo in the N9 outcome DECISIONS entry + R1 entries + R1 closure.** Lewie's N9 spike-report chat message correctly reported `2025-11-25`; I wrote `2025-11-05` in the N9 outcome and it propagated through R1 → config default → docstrings → tests. Fix: re-pin default to `2025-11-25` + correction notes on three DECISIONS entries + regression test `TestRealClaudeCodeProtocolVersion::test_shim_advertises_version_that_real_claude_code_accepts` with hardcoded 2025-11-25 expectation. Also surfaced the asymmetric client-rejection behavior: the shim's non-hostile mismatch policy (N10) keeps the SHIM alive when the client sends a different version, but doesn't help if the CLIENT rejects the shim's response — so the default must match what real Claude Code sends.

3. **Opus 4.7 `temperature` deprecation bug** (commit `5d29e3f`). Anthropic returned `HTTP 400 {'type': 'invalid_request_error', 'message': '`temperature` is deprecated for this model.'}`. Research via claude-code-guide agent (WebFetch against docs.anthropic.com) confirmed: no direct replacement parameter; Opus 4.7 is stochastic-by-default; replacement tuning knob is `output_config.effort` (one of `low`/`medium`/`high`/`xhigh`/`max`). Also uncovered the ~35% tokenizer inflation. Fix: option (C) per proposal — remove `temperature` parameter entirely from the request body, add `output_config.effort="xhigh"` (matches CLAUDE.md's project-wide effort principle), drop `recommend_temperature` config field (clean schema break), bump `recommend_max_tokens` 2048 → 4096 (round number matching original N6 "raise to 4096 if soak shows truncations" disposition), replace `RecommendResponse.temperature: float` with `RecommendResponse.effort: str`. Correction note appended to operational-first pivot entry `[2026-04-21 18:00]` — the "variance during soak must come from real input differences" language was load-bearing on a premise that Anthropic's own docs say was never true (`temperature=0` never guaranteed identical outputs). Intent preserved; mechanism re-framed (temperature-pinning → effort-pinning). Regression test `tests/test_recommend_client.py` with 7 tests — asserts `temperature` NOT in outgoing `messages.create` kwargs and `output_config={"effort": "xhigh"}` IS present. Live-smoke test also updated to exercise the new shape against real API.

4. **Shim httpx cold-start timeout bug** (commit `96d619e`). First post-fix call timed out at 30s. Service was reachable; just slow on first-call memory lookup (sentence-transformers load + ChromaDB warm-up). Shim rendered misleading `## Concierge service unavailable` error — operator chasing a "service not running" ghost when service WAS running and warming up. Fix: `DEFAULT_TIMEOUT_SECONDS` 30.0 → 90.0; new `render_service_timeout()` helper distinct from `render_service_unavailable()`; all three meta-tool handlers (recommend / request_tool / list_active) now catch `httpx.TimeoutException` separately from `ConnectError`/`NetworkError` and route to the cold-start-naming renderer. Regression tests `test_timeout_renders_cold_start_distinct_from_unavailable` (mocked timeout, asserts cold-start language + retry guidance) + `TestHttpClientColdStartTimeout::test_default_timeout_is_90_seconds` (hardcoded constant assertion). Deferred: uvicorn-startup pre-warm of sentence-transformers + ChromaDB — would shift cold-start tax off first user-visible call; soak-phase optimization, not reliability fix. Noted in `http_client.py` module docstring.
