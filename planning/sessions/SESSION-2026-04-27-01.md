## Appendix C — Discipline carry-forward: wiring-test meta-lesson, four data points

The wiring-test meta-lesson now has **four data points across two days**, all rooted in the same gap pattern: unit tests mock the wire shape (subprocess.run args, sink.publish payload, network bytes); reality has more semantics than the wire shape captures (CLI argument parsing, JSON deserialization, filesystem rename + shebang baking, persistent state semantics).

**The four data points:**

1. **Day 5 Task 0 — Telemetry-commit rollback.** Service-layer unit tests asserted "service called the sink with the right args" using an in-memory `_CapturingSink`; endpoint tests stubbed the service via `_StubService`. Neither exercised the request → real-`make_db_sink` → real-`Session` lifecycle → fresh-session-SELECT chain. The rollback-after-flush bug shipped to Day 4 close. Day 5 wiring-test (`tests/test_recommend_endpoint_persistence.py`) closed the gap. (`SESSION-2026-04-26-01.md` Task 0 close-out + Appendix A audit table)

2. **Day 5 Task 1 — SSE wire-format JSON-vs-Python-repr.** Broker-direct tests asserted "broker received the event dict with the right fields"; service-publish tests asserted "publish was called." Day 4's manual curl asserted "bytes flowed on the wire." None asserted "the bytes parse as JSON the way an HTMX client would parse them." sse-starlette's default `str()`-on-non-string serialization produced Python repr (`{'filename': '...'}`), which `JSON.parse` rejects. Day 5 wiring-test (`TestStreamingWireFormat`) used real httpx + real uvicorn-in-thread + explicit `json.loads` to catch the gap. (`SESSION-2026-04-26-01.md` Task 1 close-out)

3. **Day 6 Step 5 (correction 1) — `--system-site-packages=False` CLI rejection.** Unit tests mocked `subprocess.run` and asserted the argv shape per the (drafted) Decision B contract. The mock didn't validate argv against the actual stdlib `venv` CLI's argparse semantics. The integration test (`TestRealVenvBootstrap`) ran real `python3 -m venv` and got `argument --system-site-packages: ignored explicit argument 'False'` — the flag is `store_true`, not `key=value`. (DECISIONS Decision B post-hoc correction note, item 1)

4. **Day 6 Step 5 (correction 2) — Atomic-rename incompatible with venv shebang baking.** With correction 1 applied, the integration test surfaced a deeper issue: pip's shebang in `<venv>.tmp/bin/pip` was hard-coded to `<venv>.tmp/bin/python3.12`. After atomic rename to `<venv>/`, the shebang still pointed to the now-nonexistent `.tmp` path. Bash returned `cannot execute: required file not found` — referring to the **shebang target**, not the script. The unit-test side_effect (`_make_subprocess_run_creating_venv`) just `mkdir`'d the target and wrote a fake `bin/python` stub — no shebangs, no `pyvenv.cfg`. (DECISIONS Decision B post-hoc correction note, item 2)

**Discipline statement:** the wiring-test meta-lesson is now well-established. Future test-strategy choices should default to **"if the contract is operator-observable, the wiring test must exercise it against reality, not mocks."** Concretely:

- If the contract is a CLI invocation (argparse, argv parsing, exit codes), the wiring test runs the CLI for real
- If the contract is a filesystem operation (rename, exec, shebang interpretation), the wiring test exercises the actual filesystem
- If the contract is a network protocol (SSE wire format, JSON parsing), the wiring test runs a real client against a real server and parses with the real parser
- If the contract is a persistent state operation (DB commit, transaction lifecycle), the wiring test reads from a fresh session/transport boundary and asserts the consumer-visible state changed

Mock-based unit tests retain their value for orchestration logic (was X called Y times with Z args?); they're insufficient for any contract that touches OS-level reality. Worth landing as a peer ops-protocol section to "Live verify discipline" — flagged in Open Question 3.
