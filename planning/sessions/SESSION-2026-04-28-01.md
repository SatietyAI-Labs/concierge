## Appendix C — Venv corruption event: torch truncation diagnosis + remediation

### Event description

`tests/test_memory.py` (11 failures) and `tests/test_identity.py` (2 failures) failed with `module 'torch' has no attribute 'LongTensor'` since at least Day 5 close (`8ee0af3`). Per Days 5/6 verification, classified as pre-existing environment failure outside Concierge code; deferred to Day 7 per cut-if-behind ladder.

### Diagnosis (Day 7 Task 2 surface investigation)

Inspection revealed the torch package was **truncated**, not a version regression:

- `uv pip show torch` reports v2.11.0 installed (dist-info was complete)
- `dir(torch)` returns empty list; `torch.__version__` raises AttributeError
- Module loads as `<module 'torch' (namespace) from [...]>` with `__file__` = None — Python's PEP-420 namespace-package fallback for directories without `__init__.py`
- `torch-2.11.0.dist-info/RECORD` lists 11,678 files for the package, including `torch/__init__.py` (107,845 bytes); `find ./torch -type f | wc -l` returns 3,958 (~34% completeness)
- `torch/__init__.py` MISSING from disk despite being in RECORD

### Root cause

Truncated install — likely interrupted wheel extraction or transient disk/network failure mid-install. `uv pip install` populated dist-info before completing the package body, so pip metadata reports the install as successful. The PEP-420 namespace-package fallback masked the corruption from any `import torch` smoke check that doesn't access attributes.

### Remediation

Two-phase per Option E (try Option A first; escalate if needed):

1. **`uv sync --reinstall-package torch`** (~9m 39s) — fresh extraction. Post-install verification:
   - File count: 3,958 → **11,678** (matches RECORD exactly)
   - `torch/__init__.py` present at 107,845 bytes (matches RECORD's sha256 by size)
   - `uv run python3 -c "import torch; print(torch.__version__, torch.LongTensor)"` produces `2.11.0+cu130 <class 'torch.LongTensor'>`

2. **Mid-stream re-surface — uv-sync side effect**: the reinstall command also uninstalled 5 dev packages (pytest, pytest-asyncio, pytest-timeout, iniconfig, pluggy) because dev extras aren't in the default install set. Recovered via `uv sync --extra dev` (2.22s).

### Verification

Targeted re-run on `tests/test_memory.py` + `tests/test_identity.py`: 51/51 passing (the 13 previously-failing + 38 already-passing). Full-suite: zero new failures, zero other corruption symptoms, exactly 13 torch failures resolved.

### Operational discipline notes

**Lesson generalized**: `uv sync --reinstall-package X` strips dev deps as a side effect because dev extras are not in the default install set. Always pair with `--extra dev` (or `--all-extras`) when reinstalling individual packages: `uv sync --extra dev --reinstall-package X`. Generic uv gotcha; Concierge-specific risk because the test surface is essential and losing pytest without warning is a meaningful operational hazard for any future package reinstall. The lesson generalizes — this is the rule for any package, not just torch.

**Audit trail rationale**: the venv state isn't tracked in `git log` — `git log` between Days 6 and 7 doesn't show any artifact for this remediation. This Appendix is the future-readable record of "what was done to the venv on Day 7." The pattern (venv-state corruption events get documented in SESSION snapshot Appendices) is **flagged as candidate** for ops-protocol ratification, NOT ratified today. Per the same maturation discipline as the wiring-test discipline (which waited for four data points before ratification), one event isn't enough to commit to a convention.
